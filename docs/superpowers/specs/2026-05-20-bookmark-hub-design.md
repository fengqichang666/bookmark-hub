# Bookmark Hub 设计说明

## 1. 项目目标

在 `D:\workspace\bookmark-hub` 下创建一个团队协作型浏览器书签管理项目，包含前端和后端两部分。

第一版目标：

- 支持团队成员登录后管理团队公共书签
- 支持书签新增、查询、修改、删除
- 支持团队公共分类管理
- 支持团队成员管理
- 支持导入浏览器导出的书签 HTML 文件
- 支持基础权限控制：公共可见，只有创建者或管理员可修改、删除书签

本项目第一版聚焦“完整可用”，不追求过度扩展。

## 2. 范围与约束

### 2.1 已确认范围

- 产品形态：团队版
- 使用终端：PC 管理后台优先
- 权限规则：公共可见，只有创建者或管理员可修改、删除书签
- 导入能力：支持导入浏览器导出的书签 HTML 文件
- 技术方向：前后端分离，尽量使用较新的稳定技术栈

### 2.2 第一版不做

- 浏览器插件实时同步
- 开放注册、邀请注册链接、找回密码
- 复杂 RBAC 权限体系
- 多团队切换优化
- 标签系统
- 公开分享链接
- 回收站
- 操作审计明细
- Redis、消息队列、Elasticsearch 等额外基础设施
- 微服务拆分

## 3. 整体架构

项目采用前后端分离的单仓结构：

```text
bookmark-hub/
  frontend/
  backend/
  docs/
```

运行方式：

- 前端本地开发端口：`5173`
- 后端本地开发端口：`8080`
- MySQL 端口：`3306`

整体访问流程：

1. 用户在前端登录
2. 前端通过后端认证接口获取当前用户信息
3. 用户进入团队工作台
4. 前端通过 REST API 完成书签、分类、成员、导入记录的读写
5. 所有业务数据持久化到 MySQL

## 4. 技术栈

### 4.1 前端

- React 19
- Vite
- TypeScript
- React Router
- TanStack Query
- Zustand
- Ant Design 5

前端采用 React 全家桶组合，优先满足 PC 管理后台场景。组件库选择 Ant Design 5，是因为表格、树、上传、弹窗、表单、权限提示等后台能力成熟，能更快搭建稳定的一版。

### 4.2 后端

- Java 21
- Spring Boot 3.4.x
- Spring Web
- Spring Security
- Spring Data JPA
- Bean Validation
- Flyway
- Lombok
- MapStruct
- springdoc-openapi

后端以 REST API 为主，使用 Spring Boot 新版生态完成认证、权限、数据访问、参数校验、数据库迁移和接口文档。

### 4.3 数据库

- MySQL 8

不引入额外缓存或搜索中间件，保证第一版部署简单、调试直接。

## 5. 页面设计

第一版只做 6 个核心页面。

### 5.1 登录页

能力范围：

- 用户名密码登录
- 登录成功后进入工作台

限制：

- 不提供自助注册
- 用户由管理员预先创建

### 5.2 工作台首页

展示内容：

- 书签总数
- 分类总数
- 成员总数
- 最近新增书签
- 最近导入记录

目标是提供团队概览和导航入口，不做复杂统计分析。

### 5.3 书签管理页

能力范围：

- 书签列表展示
- 按关键词搜索
- 按分类筛选
- 查看详情
- 新增书签
- 编辑书签
- 删除书签

字段范围：

- 标题
- URL
- 描述
- 所属分类
- 可见范围
- 创建人
- 创建时间

权限规则：

- 所有团队成员可查看团队公共书签
- 创建者可编辑、删除自己的书签
- 管理员可编辑、删除任意团队书签

### 5.4 分类管理页

能力范围：

- 查看分类树
- 新增分类
- 编辑分类
- 删除分类

结构约束：

- 第一版最多支持两级分类
- 分类属于团队，不属于个人

权限规则：

- 只有管理员可管理分类
- 普通成员只能查看分类并用于筛选书签

### 5.5 团队成员页

能力范围：

- 查看成员列表
- 新增成员
- 设置成员角色
- 设置成员启用或禁用状态

角色约束：

- 只有 `ADMIN` 和 `MEMBER` 两种角色

权限规则：

- 只有管理员可管理成员

### 5.6 书签导入页

能力范围：

- 上传浏览器导出的书签 HTML 文件
- 解析并展示预览结果
- 选择导入目标分类
- 确认导入
- 查看导入结果统计

导入规则：

- 只支持标准书签 HTML 文件
- 自动解析标题和 URL
- 尽量读取文件夹结构并映射到分类
- 允许提示重复 URL
- 第一版不做复杂去重合并策略

## 6. 数据模型

第一版核心数据模型如下。

### 6.1 `team`

表示团队。

建议字段：

- `id`
- `name`
- `description`
- `created_at`
- `updated_at`

### 6.2 `user`

表示系统用户。

建议字段：

- `id`
- `username`
- `password_hash`
- `display_name`
- `email`
- `status`
- `created_at`
- `updated_at`

说明：

- 密码必须加密存储
- `status` 表示启用或禁用

### 6.3 `team_member`

表示用户和团队的关联关系。

建议字段：

- `id`
- `team_id`
- `user_id`
- `role`
- `joined_at`

说明：

- `role` 只允许 `ADMIN`、`MEMBER`

### 6.4 `category`

表示团队分类。

建议字段：

- `id`
- `team_id`
- `name`
- `parent_id`
- `created_by`
- `created_at`
- `updated_at`

说明：

- `parent_id` 用于支持两级分类

### 6.5 `bookmark`

表示书签。

建议字段：

- `id`
- `team_id`
- `category_id`
- `title`
- `url`
- `description`
- `visibility`
- `created_by`
- `created_at`
- `updated_at`

说明：

- 第一版只启用 `TEAM` 可见范围
- 保留 `visibility` 字段，方便后续扩展

### 6.6 `import_record`

表示导入记录。

建议字段：

- `id`
- `team_id`
- `operator_user_id`
- `file_name`
- `total_count`
- `success_count`
- `failed_count`
- `status`
- `created_at`

## 7. 权限规则

第一版权限规则明确如下：

1. 登录用户必须属于某个团队，才能进入系统。
2. 团队内所有成员都可以查看团队公共书签和分类。
3. 书签只有创建者或当前团队管理员可以修改、删除。
4. 分类只有管理员可以新增、编辑、删除。
5. 团队成员管理只有管理员可以操作。
6. 导入书签时，导入出的书签创建者记为当前操作人。

## 8. 导入流程

第一版导入采用两阶段流程。

### 8.1 预解析

1. 用户上传书签 HTML 文件
2. 后端解析 HTML 结构
3. 返回解析预览结果，包括文件夹和书签条目

### 8.2 确认导入

1. 用户查看预览结果
2. 用户选择导入目标分类，或按文件夹自动映射
3. 用户确认导入
4. 后端批量写入书签数据
5. 后端生成导入记录并返回结果统计

### 8.3 导入结果

结果需要至少包含：

- 总条数
- 成功条数
- 失败条数
- 失败原因摘要

## 9. 接口轮廓

### 9.1 认证接口

- `POST /api/auth/login`
- `GET /api/auth/me`

### 9.2 工作台接口

- `GET /api/dashboard/overview`

### 9.3 书签接口

- `GET /api/bookmarks`
- `GET /api/bookmarks/{id}`
- `POST /api/bookmarks`
- `PUT /api/bookmarks/{id}`
- `DELETE /api/bookmarks/{id}`

### 9.4 分类接口

- `GET /api/categories/tree`
- `POST /api/categories`
- `PUT /api/categories/{id}`
- `DELETE /api/categories/{id}`

### 9.5 团队成员接口

- `GET /api/team-members`
- `POST /api/team-members`
- `PUT /api/team-members/{id}/role`
- `PUT /api/team-members/{id}/status`

### 9.6 导入接口

- `POST /api/imports/parse`
- `POST /api/imports/confirm`
- `GET /api/imports`

- 工作台

## 11. 验收标准

### 11.1 环境验收

- 本机可运行 `Java 21`
- 本机可运行 `Maven`
- 本机可运行 `MySQL 8`
- 前端可通过 `npm install` 和 `npm run dev` 启动
- 后端可通过 `mvn spring-boot:run` 启动
- 前后端可完成本地联调

### 11.2 功能验收

- 管理员可以登录并创建成员
- 普通成员可以登录并查看团队书签
- 普通成员可以新增自己的书签
- 普通成员不能修改他人书签
- 管理员可以修改和删除任意团队书签
- 管理员可以维护分类
- 浏览器书签 HTML 可以完成预解析和确认导入
- 导入后的书签可以在列表中查询和筛选

### 11.3 数据验收

- 核心表通过数据库迁移脚本自动创建
- 用户密码非明文存储
- 所有更新和删除操作都校验团队归属与权限

### 11.4 文档验收

- 项目根目录包含启动说明
- 后端提供 OpenAPI 文档
- 系统初始化一套可登录管理员账号

## 12. 后续实现原则

实现阶段遵循以下原则：

- 仅做完成第一版所需的最小改动
- 优先保证代码清晰和可验证
- 不提前引入未来功能的抽象
- 每一步都以可启动、可联调、可验收为目标推进
